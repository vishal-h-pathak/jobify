# V3C-E1A Fill Engine Core — Implementation Plan

> Executed inline in this session (not subagent-driven-development): the
> modules are sequentially dependent on shared types designed as one unit
> (survey → maps → plan → drivers → fill), so per-task implementer/reviewer
> round-trips would mostly re-derive context already fixed here. TDD per
> module, final independent code-review pass before push.

**Goal:** Build `extension/engine` — a standalone TS package that surveys a
job-application DOM, plans field fills from a `SubmitPacket` against L0
(greenhouse/lever/ashby/workday) maps, and executes those fills with
read-back verification. Zero `chrome.*`, zero network, zero LLM.

**Spec:** `planning/session-prompts/41_v3c_e1_engine.md` (pinned contract),
`planning/V3C_DESIGN.md` §0/§2/§3.

## Global constraints
- Package root exports EXACTLY: `survey`, `planFills`, `executeFills`, and
  the pinned types — nothing else.
- No `chrome.` / `browser.` reference anywhere in `extension/engine/src`.
- Deny-test: no bundled map's fill target selector/label matches
  `/submit|apply now|send application|finish/i`.
- Drivers only ever accept `SurveyField`, never `SurveyButton` (enforced at
  the type level + a `// @ts-expect-error` constitution test).
- Alex Quinn persona only in fixtures. Scrub gate must pass.
- Diff stays entirely inside `extension/engine/`.

## Design notes worth recording (not obvious from the spec text alone)

**`FillReport.requiredEmpty` mechanism.** `FillInstruction` has no `required`
flag and `executeFills` gets no `ats`/map parameter, yet `requiredEmpty` must
reflect the *map's* required flag (parity with Python's
`apply_field_map`), not just the DOM's `required` attribute (many ATS forms
don't set it). Resolution: `planFills` keeps a module-level `WeakMap<
FillInstruction[], string[]>` keyed by the exact array it returns, recording
labels of required-but-empty-or-unmatched specs as it builds the plan.
`executeFills` reads that WeakMap for the `plan` array it was given (falling
back to `[]` if the array wasn't produced by `planFills`), then unions in any
survey field where `field.required === true` and the post-fill live value is
still empty. This keeps the pinned signatures exact while preserving Python
parity, and degrades gracefully for a hand-built plan array.

**Label ladder order (survey.ts) differs from the Python ancestor.** Python's
`get_form_fields` tries `aria-label` first; the pinned spec for this port
orders it `<label for>` → wrapping `<label>` → `aria-label`/
`aria-labelledby` → placeholder → nearest preceding text / fieldset legend.
Implement the NEW order — it's what's pinned, not what Python did.

**Checkbox/radio value encoding.** `radio_group.value` = the checked option's
label (or `""`). `checkbox.value` = `"true"` when checked, `""` otherwise —
documented in a code comment since the pinned type only spells out the
radio_group case explicitly.

**Company/Title labels never resolve.** `SubmitPacket.identity` has no
company/title fields (only contact identity). The L0 maps' "Current
Company"/"Company"/"Current Title"/"Title" labels are never `required` in
any map, so they always resolve to `""` and are silently skipped — matches
Python (`build_field_map` reads the same nonexistent keys as `""`).

**Lever full-name override** (port of `lever.py:96-99`): before joining,
override the "Name"/"Full Name" packet-derived values to
`packet.identity.full_name || \`${first} ${last}\`.trim()`.

**Settle helper.** MutationObserver-based quiescence: resolve when no
mutations fire for `QUIET_MS` (300ms), bounded by `MAX_WAIT_MS` (3000ms) —
never a global load-state wait (F1). Internal to `fill.ts`, not exported.

**React fix (F3).** Text/textarea drivers write through the native property
setter (`Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype,
'value').set.call(el, value)`) then dispatch bubbling `input`+`change`
events — plain `el.value = x` gets silently reverted by React's controlled-
input tracker. `keystrokes` strategy (escalation target) dispatches per-char
`keydown`/`keypress`/`input`/`keyup` sequences instead.

## Tasks (executed in order, one commit each)

1. Scaffold package (`package.json`, `tsconfig.json`, `vitest.config.ts`,
   `src/types.ts` with the pinned types + `SubmitPacket` copied verbatim from
   `web/lib/submit/types.ts`, `src/index.ts` stub).
2. `survey.ts` + fixtures (greenhouse, lever, ashby-like w/ shadow root,
   workday-like w/ open shadow root + typeahead, generic, same-origin
   iframe) + `survey.test.ts` (ladder rung-by-rung, shadow/iframe/kind
   detection).
3. `maps.ts` (verbatim transcription of `field_maps.yml` GH/Lever/Ashby +
   new Workday map anchored on `data-automation-id`) + `maps.test.ts`
   (table-driven parity against the YAML).
4. `plan.ts` (`build_field_map` port + Lever override + selector/label join)
   + `plan.test.ts`.
5. `drivers.ts` (text/textarea, select, checkbox, radio_group, combobox,
   file, date; native→keystrokes escalation) + `drivers.test.ts` (incl. a
   React-controlled-input fixture that reverts non-native writes).
6. `fill.ts` (`executeFills`: dispatch → read-back → retry → `FillReport`)
   + `fill.test.ts` (escalation to keystrokes, `stuckAfterReadback`).
7. `constitution.test.ts` (exports-exactly-pinned-API, no `chrome.`/
   `browser.` grep, deny-lexicon walk over bundled maps, driver type-level
   `SurveyButton` rejection).
8. Exit check: `npx vitest run` + `npx tsc --noEmit` green inside
   `extension/engine`; repo suites untouched; scrub gate PASS; independent
   code-review pass; commit; push (no merge).
