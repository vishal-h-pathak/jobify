# V3C-PACKET: application profile + submit packet (backend) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development.

**Goal:** Build the data half of V3c P0 — the encrypted `application_profiles` table +
its CRUD routes, and the `GET /api/submit/packet` endpoint that assembles everything a
future submit kit/extension needs to prefill an application. No UI, no LLM calls.

**Source of truth:** `planning/session-prompts/40_v3c_packet.md` (the session prompt —
binding) and `planning/V3C_DESIGN.md` §5 + §8 (context). The session prompt's PINNED
CONTRACT block is shared verbatim with the parallel session 39 (the kit) — implement it
exactly, do not rename fields.

  the scrub gate's forbidden tokens literally anywhere, including in comments.
- **"Every absent value is `""`, never invented."** Everywhere the `SubmitPacket`
  contract has a required `string` field whose backing data may be null/missing,
  coalesce to `""` — never omit the key, never fabricate a placeholder like "N/A".
- **Identity precedence (packet only, task 5):**
  - `email` = the authenticated user's `auth.getUser()` email, **always** — never read
    email out of `profile.yml`, even though `profile.yml`'s `identity.email` exists.
  - `first_name`/`last_name`/`full_name` — parsed from `profiles.doc["profile.yml"]`'s
    `identity.name` (a single full-name string; there is no existing first/last
    splitter in this repo — write a simple `split(" ")` on first space: everything
    before the first space is `first_name`, the rest is `last_name`, empty string if
    there's no space).
  - `phone` / `location` / `linkedin_url` / `github_url` / `portfolio_url` — the
    decrypted `ApplicationProfile.contact` field wins when present and non-empty;
    otherwise fall back to `profile.yml`'s `identity` block (`phone`, `location_base`,
    `linkedin`, `github`, `website` respectively); otherwise `""`. (This is what the
    session prompt's "contact fields from the decrypted application profile" +
    the test note "profile contact wins over blanks" specify together.)
  - There is **no existing** identity-reader helper anywhere in `web/lib` today (verified
    by search) — task 5 creates the one canonical accessor (`web/lib/submit/identity.ts`)
    so nothing re-parses `profile.yml`'s `identity` block ad hoc inline in `packet.ts`.
    Future dossier/tailor consumers can import it later; you are not modifying dossier
    or tailor code in this plan.

```ts
export function encryptJson<T>(value: T): string {
  return encryptKey(JSON.stringify(value));
}

export function decryptJson<T>(blob: string): T {
  return JSON.parse(decryptKey(blob)) as T;
}
```

### `packet.ts` — assembly

```ts
export async function buildSubmitPacket(
  admin: SupabaseClient<Database>,
  userId: string,
  authEmail: string,
  postingId: string
): Promise<
  | { ok: true; packet: SubmitPacket }
  | { ok: false; status: 409; error: "no_application_profile" }
  | { ok: false; status: 404; error: "no_materials" }
> { ... }
```

Steps, in order (each a separate `await`, bail early per the return-type branches
above):
1. `loadApplicationProfile(admin, userId)` (Task 3) → `null` → return the 409 branch.
2. Query `tailor_runs`: `.select("id, posting_id, doc_sha256, status")
   .eq("user_id", userId).eq("posting_id", postingId).eq("status", "succeeded")
   .order("created_at", {ascending: false}).limit(1).maybeSingle()` → no row → return
   the 404 branch. (This single query, scoped by both `user_id` and `posting_id`,
   is what makes "another user's run" and "no succeeded run yet" collapse to the
   identical 404 — same refusal-matrix shape as
   `app/api/tailor/materials/[runId]/route.ts`.)
3. Query `postings`: `.select("id, title, company, application_url")
   .eq("id", postingId).maybeSingle()` — throw on error (this row must exist by FK
   from `tailor_runs`; don't swallow).
4. `getProfileDoc(supabase-or-admin, userId)` — **note:** `getProfileDoc`
   (`@/lib/db/profiles.ts`) takes a `SupabaseClient<Database>` and does its own
   `.eq("user_id", userId)` — the admin client works fine here as its RLS bypass is a
   superset of the own-row policy. Pass `admin`.
5. `buildIdentity(profileDoc?.doc ?? null, applicationProfile, authEmail)` (Task 5's own
   helper above).
6. `signMaterials(admin, userId, postingId, <expiry>)` (reuse `web/lib/materials/
   signMaterials.ts` unmodified — pick the same 300-second expiry constant the tailor
   materials route uses, named the same way) for `resume.pdf`/`cover_letter.pdf` signed
   URLs (`urls["resume.pdf"] ?? ""`, `urls["cover_letter.pdf"] ?? ""`).
7. Separately download `cover_letter.txt`'s actual text: `admin.storage.from
   ("job-materials").download(\`${userId}/${postingId}/cover_letter.txt\`)` → if error
   or no data, `cover_letter_text: ""`; else `await data.text()`. (Local `const BUCKET =
   "job-materials"` in this file — `signMaterials.ts`'s own comment already notes this
   repo has no shared-const convention for that literal; don't refactor that file to
   export it, just repeat the literal here with a short comment pointing at
   `signMaterials.ts`'s precedent.)
8. `detectAtsKind(posting.application_url ?? "")` (Task 4) for `ats_kind`.
9. Assemble and return the `{ok: true, packet}` branch:
   - `posting`: `{id: posting.id, title: posting.title ?? "", company: posting.company
     ?? "", application_url: posting.application_url ?? "", ats_kind}`
   - `identity`: from step 5
   - `materials`: `{resume_pdf_url, cover_letter_pdf_url, cover_letter_text}`
   - `authorization`/`logistics`/`self_id`: straight from `applicationProfile` (each
     defaults to `{}` if the loaded profile is somehow missing that key — it won't be,
     since `sanitizeApplicationProfile` always includes all four top-level keys, but
     don't assume that invariant here without a fallback)
   - `meta`: `{tailor_run_id: run.id, doc_sha256: run.doc_sha256, generated_at: new
     Date().toISOString()}`
