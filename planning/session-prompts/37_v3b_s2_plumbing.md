## Owner decision binding this session (2026-07-06)
**PDF resume upload** joins .txt/.md everywhere a resume enters: onboarding's
evidence stage and Settings → Resume. Extraction is server-side
(`extractText.ts`): PDF → plain text, size-capped (~5MB), friendly rejection
for image-only/encrypted PDFs ("couldn't read text in this PDF — paste it
instead"), extracted text flows into the EXACT same downstream paths as
pasted text (no new persistence shape). Client upload affordances change
copy to ".pdf, .txt, or .md".

## Build (per V3B_DESIGN — condensed)
1. **`dispatchTailor`** (`web/lib/tailor/dispatchTailor.ts`) — mirrors
   dispatchHunt: authed + invite-or-admin gate upstream; inserts the
   tailor_runs row (status `requested`), dispatches `hosted-tailor.yml` with
   the design's inputs, honest error mapping (429 on the per-posting
   cooldown + 5/day count from tailor_runs, 502 dispatch failure, 503 env
   unset). `POST /api/tailor/run` wraps it.
2. **Status + materials:** `GET /api/tailor/runs?posting_id=` → the user's
   runs w/ status; `GET /api/tailor/materials/[runId]` → service-role signed
   URLs (short-lived) for resume.pdf / cover_letter.pdf / claims.json +
   honesty report — own-run access only (verify user_id server-side before
   signing). No signed URL for anything the run didn't produce.
3. **PDF ingestion** per the owner decision above — onboarding + settings
   paths, one shared helper, tests with a real small PDF fixture (checked in,
   generated by a script — no binary-document scrub-gate violation: confirm
   `scripts/scrub_gate.sh`'s tracked-binary rule and, if .pdf fixtures are
   blocked, generate the fixture at test time instead of checking it in).
4. **types.ts** additive tailor_runs entry matching the design SQL.
