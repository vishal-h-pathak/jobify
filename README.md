# jobify

A single-user, local job-application pipeline: **hunt → tailor → submit**,
plus a thin dashboard that drives the three steps with one click each.

- **Hunt** discovers and scores roles against your profile.
- **Tailor** generates a tailored resume + cover letter + form-answer drafts.
- **Submit** opens the live application with every field pre-filled and
  **stops before the final submit** — you review and click the site's own
  Submit button.

Bring your own keys (Anthropic + your own Supabase project). Nothing is
hard-coded to any one person: your personal "ground-truth" layer is a
**profile directory** generated by a one-time onboarding flow.

## Get started

→ **[docs/SETUP.md](docs/SETUP.md)** — clone → Supabase project → apply

## Layout

```
jobify/
├── jobify/                  # Python package
│   ├── hunt/  tailor/  submit/  shared/
│   └── config.py  db.py  notify.py  profile_loader.py
├── dashboard/               # Next.js cockpit (login → triage → review → runs)
├── migrations/              # Supabase schema baseline (0001_init.sql)
├── docs/                    # SETUP · ARCHITECTURE · ONBOARDING
├── .github/workflows/       # CI + optional hunt cron
├── .env.example
└── pyproject.toml
```

Three console scripts (see `pyproject.toml`):

| Script | Role |
|---|---|
| `jobify-hunt`   | discover roles, score, upsert |
| `jobify-tailor` | tailor resume / cover letter / form answers (no browser) |
| `jobify-submit` | visible-browser pre-fill, stop-at-submit |

## Development

```bash
pip install -e '.[dev]'
pytest
```

### Supabase key contract

The pipeline tables (`jobs`, `runs`, `application_attempts`) have RLS
enabled with **no policies** — an anon key gets HTTP 200 and empty result
sets, no error. jobify therefore runs **service-role only**: `jobify.db`
resolves its client from `SUPABASE_SERVICE_ROLE_KEY` (falling back to
`SUPABASE_KEY`, which must also hold the service-role key) and raises at
startup if the resolved key is demonstrably anon.

### Canonical job statuses

`jobify/shared/status.py` is the single source of truth for the
`jobs.status` enum; `jobify/shared/status.json` is its generated artifact
(regenerate with `python -m jobify.shared.status`), consumed by the
dashboard's type generator. The Postgres CHECK constraint mirrors the same
list (`jobify/migrations/0001_init.sql`). `tests/test_status_contract.py`
fails CI if any of the three drift.
